# 知识库 (LLM wiki) 读写协议

## 触发场景

任何涉及「沉淀知识」或「回忆/引用知识」的时刻：用户说「记下这个」「存进知识库」；你产出了值得长期复用的领域知识；或当前话题与 `wiki/index.md` 里某页相关。

## 知识库是什么

- **位置**：`wiki/`，是你**拥有并维护**的领域知识库（"你对世界的认知"：投研、AI、公司、技术…）。
- **边界**：只装**领域/世界知识**。个人偏好画像 → `memory/memory.md`；行为纠正/规则 → `rules/`。三者分开，别混。
- **结构**：
  - `wiki/index.md` — 常驻目录，按 `## 类目` 分组，每页一行。**已自动加载进你的上下文**，所以你随时知道库里有哪些页。
  - `wiki/log.md` — append-only 流水账。
  - `wiki/topics/<slug>.md` — 主题页（概念/实体/领域），一主题一文件。
  - `wiki/sources/<slug>.md` — ingest 进来的源摘要页。
- `uploads/` 里的原始文件**不可变**；`wiki/` 是你拥有的编译层。
- **slug 全库唯一**（topics/ 与 sources/ 不重名），用英文 kebab；中文名写进 frontmatter。index 行里源摘要页标 `·src`，主题页不标。

## 页面格式

```markdown
---
name: <kebab-slug>
description: <一行，会进 index.md>
metadata:
  type: topic | entity | source-summary
  updated: <YYYY-MM-DD>
---

正文。提炼后的知识，不是原文堆砌。相关处用 [[other-slug]] 互链。
```

## 自然引用（读取 wiki）—— 这是默认行为，不是可选项

`wiki/index.md` 已常驻你的上下文。**规则（必做，非建议）**：

> 当用户消息涉及的主题在 index.md 里有对应页（你看到 `[[X]]` 且话题与 X 相关）→ 在组织回答**之前**先 `Read wiki/topics/X.md`（或 `sources/`），把内容用进回答，并标明出处（如「（知识库 [[X]]）」）。

库里没有相关页就照常回答，别硬凑、别编。

## 捕获（写入 wiki）

**双轨触发**：

1. **显式** — 用户说「记下这个/记一下/存进知识库」→ 必写。
2. **自主（高门槛，宁缺勿滥）** — 仅当满足其一：
   - (a) 你刚产出了**可复用的综合/分析/判断**（不是简单复述）；或
   - (b) 用户消息里出现的**具名领域实体**（公司/人/技术）的**耐久事实**，且很可能再被问到。
   - **判据**：「若这段对话丢了，重建它要花 >5 分钟检索吗？」是 → 值得记。
   - **排除**：一次性问答、闲聊、时效性内容（今天天气、当前股价）。
   - **拿不准就不自动记**，改用纯文字问一句「要记进知识库吗？」等用户下条消息。
   - 例：YES = 你刚讲清「Haiku vs Sonnet 的成本/延迟取舍」；NO = 用户问「现在几点」；borderline =「某公司刚融资」→ 若用户在追踪该公司则记，否则问。

**每次捕获的动作**（轻量，当场完成，不扇出十几页）：

0. **先查重**：扫 index.md 找概念相近的条目；`ls wiki/topics/` 看有无 slug 变体。有相近页就**并进去**，别另起兄弟页。
1. 定位或新建相关 `wiki/topics/<slug>.md`，把知识**融进去**（提炼，不是原文粘贴）。
2. 更新 `wiki/index.md`：新建则在对应 `## 类目` 下加一行 `- [[slug]] — 摘要`（该类目不存在就先建 `## 类目` 标题，类目名用中文）。
3. 给 1–2 个明显相关页补 `[[链接]]`。
4. 核日期（`TZ=Asia/Shanghai date '+%Y-%m-%d'`），append 一行 `wiki/log.md`：`## [<日期>] capture | <标题>`。
5. **只提交 wiki，且仅在有改动时**：`git add wiki && (git diff --cached --quiet -- wiki || git commit -m "wiki: <标题>" -- wiki)`（绝不裹挟其它改动；无变化则静默跳过）。
6. **回报放在回复开头**（确保落在飞书卡片正文、不被折进工具抽屉）：`📝 记进了 [[slug]]`。

**完整性铁律**：写了 `topics/` 页就**必须**有对应 index 行；任何步骤中途失败，先回头补全或删掉半成品，绝不留下「有页无索引」的孤儿。

大型多页 ingest 留给 `/wiki-ingest`，不在轻量捕获里做。

## 撤回

用户在 `📝` 提示后说「撤回/删了/undo」：① 新建的页直接删、改过的页恢复上一版（`git` 有历史）；② 同步 index.md；③ append `wiki/log.md`：`## [<日期>] revert | <slug>`；④ `git commit -m "wiki: revert <slug>" -- wiki`。

## 显式操作（按需 skill）

- `/wiki <问题>` — 检索作答。
- `/wiki-lint` — 健康检查（矛盾/过期/孤页/缺链/缺页）。
- `/wiki-ingest <文件|URL>` — 把一份源 ingest 成 wiki 页。

## 为什么

让知识**编译一次、持续保鲜、复利增长**，而不是每次现查现推、或消失在聊天记录里。维护成本由你（LLM）承担——人只负责喂源、提问、判断。设计文档见 `../docs/llm-wiki-design.md`。

**红线**：捕获只能在正常回合里用 Write 完成，**绝不引入 Stop/PreToolUse hook 做自动捕获**——曾导致 agent 死锁、飞书管道瘫痪。
